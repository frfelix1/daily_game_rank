## Requirement Completeness

- [x] CHK001 — Is the initial display order of the five countries specified (random, alphabetical, fixed)? [Gap] ✓ Resolved: The API returns countries in arbitrary order (`contracts/puzzle-api.md §countries`). The client MUST display them in the order returned by the API with no additional shuffling; this is the initial order before the player reorders.
- [x] CHK002 — Is the visual representation of the "solved" state defined beyond "confirmed correct ordering displayed"? [Gap, Spec §FR-009] ✓ Resolved: Solved state = all five positions show 🟩 indicators, the ranking list is locked (non-draggable, `aria-disabled`), and the stat panel header is visually dimmed to distinguish it from the active stat. The submit button is hidden for solved stats.
- [x] CHK003 — Is the exact data schema for localStorage persistence documented (which fields must be saved and in what format)? [Gap, Spec §FR-017] ✓ Resolved: key `rankle_state_v1`; fields: `puzzleDate`, `stats[]{solved, guesses}`, `score`, `completed`; no solution/feedback stored.
- [x] CHK004 — Are requirements defined for when localStorage is unavailable or full (e.g., private browsing, storage quota exceeded)? [Gap, Spec §FR-017] ✓ Resolved: If `localStorage` is unavailable (SecurityError, QuotaExceededError), the game MUST still function for the current session with in-memory state only. No error is surfaced to the player. State simply resets on reload; no retry or warning is shown.
- [x] CHK005 — Are requirements defined for the loading state visual design (spinner, skeleton, placeholder text)? [Gap, Spec §FR-020] ✓ Resolved: Loading state = a centered spinner with the text "Loading today's puzzle…" occupying the game area. The game board and all interactive elements are not rendered until the puzzle loads successfully.
- [x] CHK006 — Are transition and animation requirements specified for the stat-reveal sequence and solved-state advancement? [Gap] ✓ Resolved: After a stat is solved, the solved state is displayed for 800ms before the next stat is revealed. No mandatory CSS animation is required; the delay is a JavaScript `setTimeout`. The next stat fades in (CSS `opacity` transition, 300ms) to draw the player's attention. Drag animations MUST use CSS `transform` only (plan.md §Performance Budget).
- [x] CHK007 — Are screen reader announcement requirements defined for guess submission feedback and stat-solved/game-complete events? [Gap, plan.md §Gate V] ✓ Resolved: An `aria-live="polite"` region (implemented as `LiveRegion.tsx`) announces: "Stat N solved!" when a stat completes, and "Game complete! Your score is X out of 150 points." when all three stats are solved. Guess feedback is communicated by the visible 🟩/🟥 indicators which have `aria-label` attributes (e.g., "correct position", "wrong position") on each feedback cell.
- [x] CHK008 — Are keyboard navigation requirements for drag-to-reorder specified beyond "KeyboardSensor active" (key bindings, focus behavior, announced state)? [Gap, plan.md §Gate V] ✓ Resolved: dnd-kit `KeyboardSensor` provides the following bindings — Space/Enter: pick up or drop the focused card; Arrow Up/Down: move the held card one position; Escape: cancel drag and return card to original position. Tab navigates between country cards and the Submit button. While a card is held, the `aria-live` region announces the card's current position (e.g., "Brazil, position 2 of 5"). Focus returns to the dropped card after placement.

## Requirement Clarity

- [x] CHK009 — Is "brief moment" (the pause between a stat being solved and the next stat being revealed) quantified with a specific duration? [Ambiguity, Spec §FR-009, Scenario 4] ✓ Resolved: 800ms delay. See CHK006.
- [x] CHK010 — Is the score display format defined (integer vs decimal, max possible value, initial value before any guesses)? [Ambiguity, Spec §FR-022] ✓ Resolved: Integer only (no decimals). Displayed as "X pts" (e.g., "0 pts" initially, "150 pts" for a perfect game). The maximum (150) is not shown inline during gameplay to avoid anchoring; it appears on the result screen as "X / 150".
- [x] CHK011 — Is "user-friendly error message" specified with required content (message copy, retry button label, any fallback behavior)? [Ambiguity, Spec §FR-020] ✓ Resolved: Error message copy: "Couldn't load today's puzzle. Check your connection and try again." Retry button label: "Try Again". On retry, the loading state reappears and the fetch is re-issued with the same UTC date parameter. No offline fallback puzzle is shown.
- [x] CHK012 — Is the exponential penalty formula documented with sufficient precision (base, exponent, per-guess vs per-misplaced-country) to unblock implementation? [Ambiguity, Spec §FR-011] ✓ Resolved: linear MVP formula `max(10 − 2n, 0)` per position; max 150 pts.
- [x] CHK013 — Is "usable without requiring instructions" (SC-002) defined with measurable acceptance criteria beyond subjective assessment? [Measurability, Spec §SC-002] ✓ Resolved: Accepted as a design intent for MVP. Measurable proxy for usability testing: in an informal 5-person first-use test, ≥ 4 participants complete a full game (all three stats solved) without asking for help or consulting external resources. Not a blocking gate for initial release.
- [x] CHK014 — Is "comprehensible to a general audience" (SC-007) defined with a testable method (e.g., reading level target, user test threshold)? [Measurability, Spec §SC-007] ✓ Resolved: Authoring constraint — all tooltip text MUST target a Flesch-Kincaid reading level of Grade 8 or below (Flesch Reading Ease score ≥ 60). This is enforced at puzzle authoring time, not at runtime. No automated tooling is required for MVP.
- [x] CHK015 — Is the share text format fully specified (line structure, exact emoji choices for bull/miss, score line formatting, header/footer)? [Clarity, Spec §FR-013] ✓ Resolved: header `Rankle #N — X pts`, one line per stat, guesses separated by ` / `, 🟩 = correct, 🟥 = wrong; anonymous stat labels; no country names.

## Requirement Consistency

- [x] CHK016 — Does the player-facing stat direction label requirement in FR-004 ("Rank from most to least") map unambiguously to the API contract `direction` field (`"asc"`/`"desc"`)? [Consistency, Spec §FR-004, contracts/puzzle-api.md §direction] ✓ Resolved: `"desc"` (rank 1 = highest value) → player label "Rank from most [stat] to least [stat]". `"asc"` (rank 1 = lowest value) → player label "Rank from least [stat] to most [stat]". The label is generated client-side by interpolating the stat's `label` field into the direction template. No conflict.
- [x] CHK017 — Is the "no directional hint for incorrect positions" rule (FR-008) consistent with the information visible in the shared emoji grid (FR-013) — does the grid inadvertently reveal ordering information? [Consistency, Spec §FR-008, FR-013] ✓ Resolved: No conflict. FR-008 applies during active gameplay (no hints for wrong positions while the stat is unsolved). FR-013's emoji grid is only shown post-game, is keyed by position (not by country name), uses anonymous stat labels, and includes no country names — so it conveys correctness but not direction. A recipient who didn't play cannot infer the correct ordering from the grid.
- [x] CHK018 — Does FR-021 (no past puzzles accessible) align with FR-017 (local state persists across reloads) — is it specified that only today's state is surfaced to the player? [Consistency, Spec §FR-017, FR-021] ✓ Resolved: Consistent. `rankle_state_v1` stores exactly one puzzle's state at a time. If `puzzleDate` in localStorage differs from the current UTC date, state is silently discarded and the new puzzle loads fresh (see Clarifications §CHK039). No past states are ever surfaced to the player.
- [x] CHK019 — Do the User Story 4 acceptance scenarios (daily rotation) align with the server-side date handling defined in the API contract (UTC date parameter, 404 for future dates)? [Consistency, Spec §US-4, contracts/puzzle-api.md] ✓ Resolved: Consistent. The client always derives today's UTC date via `new Date().toISOString().slice(0, 10)` and passes it to the API. US-4 Scenario 2 ("new puzzle presented on calendar day change") is fulfilled because the stored `puzzleDate` will not match the new UTC date, causing a fresh fetch. The API's 404 for future/unavailable dates is handled by the error+retry path (FR-020).

## Acceptance Criteria Quality

- [x] CHK020 — Are explicit acceptance scenarios defined for FR-020 (loading state display and error-with-retry behavior)? [Completeness, Spec §FR-020] ✓ Resolved: Three scenarios added: (1) Given the game loads, When the puzzle fetch is in flight, Then a centered spinner and "Loading today's puzzle…" are displayed and no interactive elements are rendered. (2) Given the fetch fails (network error, 4xx, 5xx, or timeout), Then the spinner is replaced by the error message (CHK011) and a "Try Again" button; the game board is still not rendered. (3) Given the player taps "Try Again", When the retry fetch succeeds, Then the loading state reappears briefly then the game loads normally.
- [x] CHK021 — Is there a worked scoring example or formula reference that makes SC-005 ("feedback unambiguously communicates correct positions") objectively verifiable? [Measurability, Spec §SC-005, FR-011] ✓ Resolved: Worked example — Stat 1, Position 3: wrong on guess 1, wrong on guess 2, correct on guess 3 → n=2, score = max(10−4, 0) = 6 pts. Position 1: correct on first guess → n=0, score = 10 pts. Position 5: never wrong → n=0, score = 10 pts. A player who solves every position on guess 1 scores 5 × 3 × 10 = 150 pts. A player who takes 6+ wrong guesses on a position scores 0 for that position (penalty floor).
- [x] CHK022 — Is SC-004 ("90% of players complete all three stats") tied to a measurement mechanism, baseline, and time window? [Measurability, Spec §SC-004] ✓ Resolved: SC-004 is a post-MVP tracked KPI. For MVP, it serves as a design quality target only. Measurement requires anonymous analytics events (`game_started`, `game_completed`) — adding these is a post-MVP task. No measurement tooling is required before initial launch.
- [x] CHK023 — Is SC-006 ("no puzzle repeats within 30 days") enforced by a documented authoring constraint or tooling check rather than relying on editorial memory? [Measurability, Spec §SC-006] ✓ Resolved: Authoring checklist step added to `contracts/puzzle-api.md` (step 6): before publishing a puzzle, verify that no puzzle file in the preceding 30 calendar days uses the same set of 5 country IDs or any of the same 3 stat labels. For MVP this is a manual check; a lint script (`scripts/check-puzzle-uniqueness.ts`) is a recommended post-MVP addition.

## Scenario Coverage

- [x] CHK024 — Are requirements defined for a player solving a stat on the very first guess (zero-penalty path and immediate stat advancement)? [Coverage] ✓ Resolved: If all 5 positions are correct on the first submission (n=0 for every position), each position scores 10 pts; the stat is solved immediately on that guess. The 800ms pause (CHK006) still applies before the next stat is revealed. No special first-guess visual treatment is required beyond the normal solved state. Acceptance scenario: Given a player submits the correct ordering on their first attempt, Then all 5 feedback cells show 🟩, the score increases by 50 pts, and the stat advances after 800ms.
- [x] CHK025 — Are requirements defined for keyboard-only navigation through the complete game flow (ranking, submitting, advancing stats, accessing result card)? [Coverage, Gap] ✓ Resolved: Full keyboard path — Tab to focus a country card → Space/Enter to pick it up → Arrow Up/Down to move it → Space/Enter to drop it → Tab to reach the Submit button → Enter to submit → (repeat for each stat) → Tab to reach the Share button on the result card → Enter to activate share/copy. All interactive elements MUST be reachable and operable via keyboard alone. Focus MUST NOT be lost or trapped at any point in the flow.
- [x] CHK026 — Are share requirements differentiated between iOS/Android native share sheet and desktop clipboard fallback, including failure handling for each? [Coverage, Spec §FR-014] ✓ Resolved: (1) If `navigator.share` is available (typically mobile), invoke native share sheet with the result text. (2) If `navigator.share` is unavailable, use `navigator.clipboard.writeText()`. (3) If clipboard write fails (permissions denied, insecure context), display the share text in a modal/textarea for manual copy and prompt the player to copy it. Confirmation feedback: a "Copied!" toast appears for 2 seconds after a successful clipboard write.
- [x] CHK027 — Are requirements defined for a player who opens the game simultaneously in multiple browser tabs (state conflict, duplicate session)? [Coverage, Gap] ✓ Resolved: Last-write-wins on `localStorage`. No cross-tab locking is implemented for MVP. If the player completes the game in tab A, tab B's next interaction reads the updated `rankle_state_v1` and renders the completed result (FR-018 path). Simultaneous active play in multiple tabs is an unsupported edge case; no special conflict resolution is required.

## Edge Case Coverage

- [x] CHK028 — Is behavior defined when the API response `date` field does not match the `date` parameter the client sent (e.g., stale CDN response)? [Edge Case, Gap, contracts/puzzle-api.md] ✓ Resolved: If `response.date` ≠ requested date, the client MUST discard the response and retry once with a `Cache-Control: no-cache` header to bypass stale CDN. If the retry also returns a mismatched `date`, the client falls back to the standard error+retry state (CHK011). This guard is implemented in the puzzle-fetch utility in `src/lib/puzzle.ts`.
- [x] CHK029 — Is behavior specified when a national flag SVG asset fails to load (broken image fallback, impact on drag interaction)? [Edge Case, Gap, Spec §FR-002] ✓ Resolved: If a flag SVG fails to load, the `<img>` element is replaced by a neutral grey rectangle (CSS background) with the country's ISO alpha-2 code as text (e.g., "BR"). The country card remains fully draggable. This is implemented via the `onError` handler on the flag `<img>` element.
- [x] CHK030 — Is behavior defined when the puzzle API returns a structurally valid HTTP 200 but a malformed or schema-invalid JSON body? [Edge Case, Gap, contracts/puzzle-api.md] ✓ Resolved: The client MUST validate the parsed response against the expected schema (presence and type of `date`, `countries` array of 5, `stats` array of 3 with required fields). Any parse failure or schema violation MUST be treated as a retrieval failure: show the error message and "Try Again" button (CHK011 / FR-020). Validation is implemented in the puzzle-fetch utility.
- [x] CHK031 — Is behavior specified for a player whose local device clock is set to a different calendar day than UTC (early or late by one day)? [Edge Case, Spec §FR-001, Assumptions] ✓ Resolved: The client derives the UTC date via `new Date().toISOString().slice(0, 10)`, which is always UTC regardless of the device's local timezone. Device clock skew (wrong absolute time) is not compensated for; the client trusts the device's reported time. A player with a significantly wrong system clock may see a puzzle for a wrong date — this is an accepted MVP limitation. Server-side date enforcement is out of scope.

## Non-Functional Requirements

- [x] CHK032 — Is the TTI ≤ 1.5s performance budget tied to specific measurement conditions (device class, network profile, measurement tool)? [Clarity, plan.md §Performance Goals] ✓ Resolved: TTI ≤ 1.5s measured via Lighthouse CI on simulated Fast 4G (≈ 9 Mbps down, 1.5 Mbps up, 40ms RTT) with 4× CPU throttle (mid-range mobile). Measurement tool: Lighthouse CI in the CI pipeline on every PR. Target applies to the initial puzzle-load page.
- [x] CHK033 — Is the 60fps drag animation requirement tied to a measurable method and minimum target device class? [Measurability, plan.md §Performance Goals] ✓ Resolved: Drag animations MUST use only CSS `transform: translate()` (no `top`/`left`/`width` mutations during drag — enforced by plan.md §Performance Budget). Verified by: Chrome DevTools Performance panel showing no layout or paint records on the main thread during drag. Target: ≤ 5 dropped frames (below 55fps) during a 2-second drag on a mid-range mobile device (4× CPU throttle). Failing this target is a build-blocking concern, not a merge blocker for MVP.
- [x] CHK034 — Are WCAG 2.1 AA color contrast requirements specified for all UI states: bull/miss indicators, score display, solved state, error state? [Completeness, plan.md §Gate V] ✓ Resolved: Contrast ratio ≥ 4.5:1 for all text (score, stat label, tooltip, error message, country name). UI component contrast ≥ 3:1 (card borders, button outlines). 🟩/🟥 emoji are Unicode characters and exempt from contrast requirements per WCAG 2.1 Exception for decorative images; however, each feedback cell MUST also carry an `aria-label` ("correct position" / "wrong position") for screen reader users. Verified via `axe-core` in component tests.
- [x] CHK035 — Are requirements defined for gameplay behavior on a degraded or intermittent connection beyond "progress preserved locally" (e.g., can the player still submit guesses offline)? [Completeness, Spec §Edge Cases] ✓ Resolved: Once the puzzle is loaded into memory for the session, all gameplay (submitting guesses, computing feedback, updating score, advancing stats) is fully functional without a network connection. Only the initial puzzle fetch requires network access. On page reload while offline, the solution re-fetch fails and the error+retry state is shown; existing guesses and score in `rankle_state_v1` are preserved and restored once connectivity is regained and the retry succeeds.
