# Research: Dynamic Puzzle Generation

**Feature**: 005-dynamic-puzzle-generation
**Phase**: 0 — Research

---

## Decision 1: Caching Strategy for the Puzzle API Route

**Decision**: Use `unstable_cache` from `next/cache` to cache generated puzzles per date with `revalidate: 86400` (24 hours), combined with the existing `Cache-Control` response header.

**Rationale**:
- `unstable_cache` is the stable, battle-tested option that works without any opt-in config changes. The new `use cache` directive requires adding `cacheComponents: true` to `next.config.ts` and is still documented as a Cache Components feature requiring explicit opt-in in Next.js 16.
- With `unstable_cache`, the puzzle is generated once per date per deployment, cached server-side, and served instantly on all subsequent requests for the same date. This satisfies the Constitution's p95 ≤ 200ms constraint.
- The cache key is the date string (e.g., `"puzzle-2026-05-27"`), so different dates never collide and yesterday's cache does not need invalidation — it naturally expires via `revalidate: 86400`.
- The existing `Cache-Control: public, max-age=86400, s-maxage=86400, stale-while-revalidate=3600` header is retained on the response, so CDN/edge caching continues to work exactly as before.

**Alternatives considered**:
- `use cache` directive (Next.js 16): Requires `cacheComponents: true` config opt-in. Cleaner future direction but adds deployment risk for an existing production app. Deferred to a future upgrade.
- In-memory module-level cache (plain `Map`): Would work within a single serverless function instance but is ephemeral — lost on cold starts and not shared across Vercel function instances. Insufficient as the sole caching mechanism.
- Route segment `export const revalidate = 86400`: Cannot be used in a Route Handler that reads dynamic query parameters (`?date=`), because the segment-level revalidate applies to the whole route, not per-parameter responses.
- External cache (Redis, KV): Adds infrastructure cost and complexity. Not needed given `unstable_cache`'s effectiveness for this use case.

## Decision 2: Deterministic Pseudo-Random Number Generator (PRNG)

**Decision**: Implement a standalone Mulberry32 seeded PRNG in `src/lib/seeded-random.ts`. The seed for any date is `BASE_SEED + dayOffset` where `BASE_SEED = 42` and `dayOffset` uses the same epoch calculation as `getPuzzleNumberForDate`.

**Rationale**:
- The Python generation script uses `random.Random(seed + day_offset)` (Python's Mersenne Twister). This cannot be replicated exactly in JavaScript/TypeScript, but that is acceptable: the puzzle files already in `data/puzzles/` were generated by Python and will be removed. The TypeScript generator needs internal consistency (same TypeScript seed → same puzzle), not cross-language consistency with the Python scripts.
- Mulberry32 is a well-known, minimal, high-quality 32-bit seeded PRNG. It has no external dependencies, produces uniform distributions, and is trivially implementable in ~10 lines of TypeScript.
- It is a pure function (state passed explicitly), satisfying Constitution Principle IV (all `src/lib/` must be pure).
- Alternative: `seedrandom` npm package. Works well but adds a dependency for something easily implemented inline.
- Alternative: xorshift32. Also viable; Mulberry32 is slightly preferred for statistical quality.

**Seed formula**: `seed = 42 + getPuzzleNumberForDate(dateStr)`. This mirrors the Python CLI default (`--seed 42`) and produces a distinct, deterministic seed per calendar day.

**Retry seeding**: When a generated candidate is rejected (constraint violation or consecutive-day duplicate), the retry uses `seed + attempt` (matching Python's `seed + i` retry increment).

## Decision 3: Dataset Loading Strategy

**Decision**: Load `data/dataset.json` at runtime using `fs.readFileSync(path.join(process.cwd(), 'data', 'dataset.json'), 'utf-8')` inside the API route, wrapped in `unstable_cache` so it is parsed only once per deployment.

**Rationale**:
- Vercel Node.js serverless functions have access to the filesystem at the path returned by `process.cwd()`, which is the project root. Files in the `data/` directory are included in the deployment bundle when they are not excluded by `.vercelignore` or `.gitignore`. `data/dataset.json` is already committed and not ignored.
- Using `fs.readFileSync` (rather than a static `import`) avoids bundling a ~150KB JSON blob into the JavaScript module graph, which would slow cold-start module parsing.
- `unstable_cache` with a permanent revalidation (no `revalidate` set, or `revalidate: false`) ensures the dataset is read and parsed exactly once per function instance lifetime — subsequent puzzle generation calls reuse the cached parsed dataset object.
- Alternative: `import dataset from '../../../data/dataset.json'` (static JSON import). TypeScript supports this (`resolveJsonModule: true`). Simpler code, but makes the JSON part of the JS bundle. For a 150KB file this is borderline acceptable; the filesystem approach is cleaner for operational separation.
- Alternative: Embed the dataset as an environment variable or fetch from an external URL. Unnecessarily complex; the file is already in the repo.

## Decision 4: Puzzle Generation Code Location and Structure

**Decision**: The generation logic lives in two new `src/lib/` files:
- `src/lib/seeded-random.ts` — pure PRNG utilities (seed, shuffle, pick)
- `src/lib/puzzle-generator.ts` — pure `generatePuzzle(dateStr: string, dataset: Dataset): PuzzleFile` function

The API route (`src/app/api/puzzle/route.ts`) is the only file that performs I/O (loading dataset, caching). It calls into `puzzle-generator.ts` with the pre-loaded dataset.

**Rationale**:
- Constitution Principle IV requires all `src/lib/` code to be pure functions with no side effects. Separating I/O (route) from pure logic (lib) is mandatory.
- Placing generation logic in `src/lib/` makes it fully unit-testable without HTTP or filesystem setup.
- The `Dataset` type (the parsed `dataset.json` shape) belongs in `src/types/index.ts` alongside `PuzzleFile` and other domain types.
- The consecutive-day uniqueness check requires generating the previous day's puzzle. Since `generatePuzzle` is pure and deterministic, calling it for `date - 1 day` is cheap and correct.

## Decision 5: Handling the `GET /api/puzzles` Route

**Decision**: Replace the filesystem-based date-listing endpoint with a computed endpoint that returns a sorted list of valid puzzle dates from epoch to today (UTC).

**Rationale**:
- The current `GET /api/puzzles` route scans `data/puzzles/` for JSON filenames. Once those files are removed, it returns an empty list.
- The dev seed-switcher panel (the only consumer of this endpoint) only needs a list of dates. Any date from epoch (`2026-01-01`) through today is a valid puzzle date by definition — the generator produces a puzzle for every date.
- Returning all dates since epoch is simple, deterministic, and requires no filesystem access.
- The response shape stays the same (array of date strings), so the dev panel needs no changes.

## Decision 6: Fate of the Existing Python Scripts and Puzzle Files

**Decision**: The Python generation scripts (`scripts/generate_puzzles.py`, `scripts/build_dataset.py`, etc.) and all pre-generated `data/puzzles/*.json` files are removed as part of this feature. The `data/dataset.json` file is retained.

**Rationale**:
- The puzzle JSON files are no longer needed — the API generates puzzles on demand.
- Keeping the Python scripts alongside a TypeScript generator creates two sources of truth with different outputs, which is confusing and error-prone.
- `data/dataset.json` is retained because it is the dataset source and is built from the raw CSV files. If the dataset needs updating, operators re-run `build_dataset.py` and commit the new `dataset.json`. This is an infrequent operation unrelated to daily puzzle generation.
- The raw data CSVs and stat-processing scripts (`normalize_stats.py`, etc.) may be retained for dataset maintenance but are not part of this feature's scope.

## Decision 7: Constitution Compliance Approach

**Decision**: All generation logic is subject to the full constitution. Specific commitments:

| Principle | How Addressed |
|---|---|
| I. TypeScript Strict Mode | `src/lib/puzzle-generator.ts` and `src/lib/seeded-random.ts` use explicit parameter and return types; no `any`; all Dataset fields typed in `src/types/index.ts` |
| II. Test-First Development | Unit tests for `puzzle-generator.ts` and `seeded-random.ts` written before implementation; integration test for the updated API route updated before the route is modified |
| III. Next.js App Router Discipline | API routes remain GET-only; `unstable_cache` used for server-side caching; no `use client` boundary changes |
| IV. Game Logic Purity | `generatePuzzle()` is a pure function in `src/lib/`; dataset loading remains in the route handler |
| V. Accessibility Baseline | No UI changes; not applicable |

**Performance**: `unstable_cache` ensures puzzle generation runs at most once per date per deployment. Subsequent requests are cache hits served from memory in <1ms. Cold-start first-generation time (running the algorithm up to 20×10 attempts) is bounded by the dataset size and algorithm complexity — acceptable within the 200ms p95 budget given the dataset is ~150KB and the algorithm is O(n log n) per attempt.
